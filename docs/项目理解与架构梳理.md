# FlipACoin 项目理解与架构梳理

## 1. 当前项目状态

- 这是一个“新玩法 + 旧框架复用”的项目。
- 玩法策划已经切到 `Flip A Coin`，见 `docs/FlipACoin_Roblox多人化改造策划案.md`。
- 代码框架仍然沿用你之前的 `SystemMgr + Systems` 架构，仓库里保留了大量旧系统，但目前在 `SystemMgr.lua` 中真正启用的系统很少。
- 当前实际启用系统：
  - `AnimateSystem`
  - `CharacterSystem`
  - `GuiSystem`
  - `MusicSystem`
  - `PlayerSystem`
- 优先加载顺序：
  - `PlayerSystem`
  - `CharacterSystem`

这说明当前工程的重点不是“功能很多”，而是先把基础运行骨架、双端同步、玩家数据、角色展示和 UI 底座搭起来。

## 2. 项目目录和职责

### 2.1 Roblox 映射

`default.project.json` 的映射很清晰：

- `src/ReplicatedFirst`
  - 最早执行的客户端加载逻辑
- `src/ReplicatedStorage`
  - 双端共享代码
  - `Systems/` 是系统框架核心
  - `configs/` 是常量和数据结构
  - `modules/` 是通用工具模块
- `src/ServerScriptService`
  - 服务端启动入口
- `src/ServerStorage`
  - 服务端专属类、数据服务和存档
- `src/StarterPlayer`
  - 玩家加入后执行的客户端逻辑
- `src/StarterGui`
  - 主 UI 和 UI 控制器

### 2.2 代码组织风格

你的系统基本采用这种目录结构：

- `Systems/<SystemName>/init.lua`
- `Systems/<SystemName>/Presets.lua`
- `Systems/<SystemName>/ui.lua`
- `Systems/<SystemName>/Assets/`

这套结构的含义很明确：

- `init.lua` 放双端系统逻辑
- `Presets.lua` 放系统配置
- `ui.lua` 放客户端 UI 层
- `Assets/` 放该系统专属资源

## 3. 启动流程

## 3.1 客户端启动链

### 最早阶段

- `src/ReplicatedFirst/Loading.client.lua`
  - 移除默认加载页
  - 挂载 `RobStar` 加载界面到 `PlayerGui`

### 玩家脚本阶段

- `src/StarterPlayer/StarterPlayerScripts/client.client.lua`
  - 只做一件事：`require(Replicated.Systems.SystemMgr)` 后执行 `SystemMgr.Start()`

### 主 UI 阶段

- `src/StarterGui/Main/uiClient.client.lua`
  - 关闭系统背包
  - 根据触屏 / 手柄做 UI 适配
  - 将主界面的按钮统一绑定到 `uiController`
  - 关闭默认重置按钮

客户端的启动思路很干净：

1. 先显示加载页
2. 再启动系统管理器
3. 再让主 UI 统一接管交互

## 3.2 服务端启动链

- `src/ServerScriptService/server.server.lua`

启动顺序：

1. `SystemMgr.Start()`
2. `GlobalDataModule.Init()`
3. 初始化 `BillboardManager`
4. 用 `ScheduleModule.AddSchedule(60, ...)` 每分钟刷新一次排行榜展示数据

可以看出你偏好把“系统框架启动”和“服务端额外业务脚本”分开：

- 通用玩法逻辑归系统
- 排行榜、全局数据、运营向脚本放在 `ServerScriptService` / `ServerStorage`

## 4. SystemMgr 的核心作用

`src/ReplicatedStorage/Systems/SystemMgr.lua` 是整个框架中最关键的文件。

它做了 5 件事：

### 4.1 管理系统注册

`systems = { ... }` 决定哪些系统真正参与本局游戏。

- 仓库里虽然有很多系统目录
- 但没有在 `SystemMgr` 注册的系统，等于没有启用

### 4.2 构造双端代理

在系统加载时，它会自动给系统创建代理方法：

- 服务端给系统生成 `self.Client` 和可选 `self.AllClients`
- 客户端给系统生成 `self.Server`

所以业务层调用会写成：

- 服务端发给某个玩家：`self.Client:Foo(player, args)`
- 服务端广播：`self.AllClients:Foo(args)`
- 客户端请求服务端：`self.Server:Foo(args)`

业务代码不直接自己 new RemoteEvent，而是统一走 `SystemMgr` 自动代理。

### 4.3 自动远程路由

所有跨端调用最终都会被编码成：

- 参数列表
- 一个尾部信息表 `{ sysName, funName }`

然后在 `RemoteEvent` / `UnreliableRemoteEvent` 回调里动态分发到目标系统方法。

这意味着你的架构本质上是：

- “系统级 RPC 路由器”
- 不是“一堆分散的 RemoteEvent”

### 4.4 白名单控制

每个系统都有 `whiteList`。

它在当前框架中的语义是：

- 这些函数不会被自动暴露成远程可调用方法
- 客户端也不能通过路由直接打到这些函数
- 更适合放服务端内部函数 / 本地辅助函数

例如：

- `PlayerSystem.GetPlayerData`
- `PlayerSystem.UpdateLeaderStats`
- `AnimateSystem.GetAnimationTrack`

### 4.5 生命周期编排

`SystemMgr` 统一管理：

- `PlayerAdded`
- `PlayerRemoving`

尤其是 `PlayerRemoving` 的顺序被你专门做了编排：

1. 先跑各系统的 `PlayerRemoving`
2. 再释放档案 `DataMgr:ReleaseProfile(player)`
3. 再删掉 `PlayerServerClass` 实例

这个顺序很重要，因为你明确要求：

- 系统清理时数据仍可写
- Profile release 必须发生在系统清理之后

这也是你 2026 年版本框架里最成熟的一点。

## 5. 客户端 / 服务端交互逻辑

## 5.1 服务端如何调用客户端

服务端系统里，普通函数会被 `SystemMgr` 自动映射到 `self.Client`。

例如 `PlayerSystem:PlayerAdded` 服务端分支会：

- 读取 `PlayerServerClass` 中的完整玩家数据
- 组装 `args.data`
- 调用 `self.Client:PlayerAdded(player, args)`

客户端收到后进入同名函数的客户端分支执行。

## 5.2 客户端如何调用服务端

客户端系统里，普通函数会被自动映射到 `self.Server`。

典型调用方式是：

- `self.Server:UsePotion(args)`
- `self.Server:ChooseInfLevel(args)`

然后由 `SystemMgr` 在服务端解包，转发到系统同名方法。

## 5.3 服务端安全模型

你的安全模型不是完全依赖 RemoteEvent 来源，而是双保险：

### 第一层

`SystemMgr` 服务端收到远程后，先校验：

- `player` 是否还在 `Players` 里

### 第二层

绝大多数系统函数内部再手动校验：

- `if sender ~= SENDER then return end`

也就是：

- 系统内部调用使用可信 token `SENDER`
- 客户端远程请求不会天然带这个 token
- 这样能区分“系统内链路”和“客户端主动请求”

这是你系统框架里非常鲜明的设计习惯。

## 5.4 白名单与安全的关系

`whiteList` 进一步限制了哪些函数不会暴露为跨端接口。

也就是说你现在的安全思路是：

1. 默认系统函数可被跨端路由
2. 白名单函数禁止远程暴露
3. 关键服务端逻辑再用 `SENDER` 校验
4. 玩家离场时做 `player alive guard`

## 6. 玩家数据链路

玩家数据的链路现在非常清楚：

### 6.1 数据源

- `src/ReplicatedStorage/configs/DefaultData.lua`
  - 定义正式默认档案结构
- `src/ReplicatedStorage/configs/DebugData.lua`
  - Studio Debug 时可覆盖默认档案

### 6.2 服务端存档层

- `src/ServerStorage/modules/DataManager.lua`
  - 基于 `ProfileService`
  - 负责加载、读取、写入、重置、释放 profile

特点：

- `ProfileStore("PlayerData", DEFAULT_DATA)`
- 玩家加入时异步加载 profile
- 加载完后给玩家挂一个 `profileLoaded` 标记
- 玩家离开时不在这里直接处理，而是交给 `SystemMgr` 编排
- release 前做 snapshot，保证短时间内还能只读访问离场数据

### 6.3 服务端玩家实例层

- `src/ServerStorage/classes/PlayerServerClass.lua`

这是你对 `DataManager` 的 OOP 包装层。

职责：

- 等待 `profileLoaded`
- 提供 `GetOneData / SetOneData / AddOneData`
- 绑定玩家 CharacterAdded / Chatted
- 承接 debug chat command

可以理解为：

- `DataManager` 负责“存档”
- `PlayerServerClass` 负责“玩家对象”

### 6.4 客户端镜像层

- `src/ReplicatedStorage/Systems/ClientData.lua`

客户端数据不是实时 DataStore，而是：

- 服务端 `PlayerSystem` 首次下发完整数据
- 客户端 `ClientData.InitData(args.data)` 初始化镜像
- 后续系统通过局部更新同步数据

这是标准的：

- 服务端权威
- 客户端缓存镜像

## 7. 当前已启用系统的职责

## 7.1 PlayerSystem

当前最关键的业务系统。

### 服务端职责

- 玩家加入后取 `PlayerServerClass`
- 发欢迎徽章
- 初始化创建时间
- 上报 onboarding 埋点
- 创建 `leaderstats`
- 下发完整玩家数据到客户端

### 客户端职责

- 初始化 `ClientData`
- 加载 `script.ui`
- 回放 UI pending calls

### 数据同步职责

- `AddExp`
- `Rebirth`
- `UpdateLeaderStats`
- `UpdatePlayerHeadGui`

这个系统本质上是：

- “玩家基础数据初始化系统”
- 也是客户端数据镜像的入口系统

## 7.2 CharacterSystem

当前负责玩家角色展示和角色基础配置。

### 服务端职责

- 玩家角色出生时克隆 `StarterGui.Templates.onPlayerHead`
- 挂到角色 `Head`
- 隐藏 Roblox 默认名字显示
- 更新头顶信息
- 设置碰撞组

### 客户端职责

- 初始化角色侧 UI

这个系统说明你喜欢把：

- 角色实体处理
- 头顶展示
- 角色碰撞

统一放在 CharacterSystem，而不是散在多个脚本里。

## 7.3 GuiSystem

职责非常单纯：

- 作为“服务端驱动客户端 UI 通知”的桥接系统

例如：

- 服务器 `GuiSystem:SetNotification(...)`
- 客户端最终调用 `uiController.SetNotification(args)`

这个系统体现了你的风格：

- 服务端不直接碰 `PlayerGui`
- 统一通过系统代理让客户端展示 UI

## 7.4 MusicSystem

负责：

- 播放 3D 音效
- 播放 2D 音效
- 广播公共音效
- 调整 bgm / sfx / weather 音量

当前状态需要注意：

- 代码里依赖 `SystemMgr.systems.SettingsSystem:GetGlobalSoundVolumeFactor()`
- 但 `SettingsSystem` 目前没有在 `SystemMgr` 启用

这说明：

- 当前 MusicSystem 是从旧项目复用来的
- 依赖链还没有完全为新项目裁剪干净

不过由于 `SystemMgr` 对未加载系统有 safe proxy，这类问题暂时不会直接炸死，而是发 warning。

## 7.5 AnimateSystem

负责统一动画播放：

- 服务端 / 客户端共用动画逻辑
- 支持玩家和 NPC/Model 两种 actor
- 用 `animTracks` 做缓存
- 支持 `PlayAnimation / StopAnimation / StopAllAnimation`

这个系统的特点是：

- 你倾向于做“中央动画系统”
- 而不是每个系统各自管理动画

## 8. UI 架构与交互方式

## 8.1 主 UI 的统一入口

`StarterGui/Main/uiClient.client.lua` 负责：

- 遍历主按钮
- 按命名约定 `xxxButton -> 打开 xxx Frame`
- 每个关闭按钮统一接入 `uiController.CloseFrame`

你明显在追求：

- “UI 行为统一”
- “按钮逻辑不要散在各个 LocalScript 里”

## 8.2 uiController 是中心控制器

`src/StarterGui/Main/uiController.lua` 负责大量 UI 通用行为：

- `SetNotification`
- `OpenFrame / CloseFrame`
- `OpenModal`
- `SetButtonHoverAndClick`
- `SetGuideButton`
- `SetHoverFrame`
- `SetOneLineTip`
- `AddTimerLabel`
- `AddReward`

也就是说：

- 系统 UI 模块负责业务
- `uiController` 负责通用动效、交互和组件控制

## 8.3 pendingCalls 模式

这是你的 UI 框架里很典型的一点。

做法是：

1. 系统客户端先创建一个假的 `Ui = { pendingCalls = {} }`
2. `setmetatable(Ui, Types.mt)`
3. 如果 UI 还没真正 require，任何调用都会被记录进 `pendingCalls`
4. 等 `PlayerAdded` 时真正加载 `script.ui`
5. 再回放之前积压的调用

这个模式解决了：

- 系统逻辑先到
- UI 模块后加载
- 调用顺序不炸

你对异步初始化顺序问题是有固定解法的。

## 10. 你的代码书写习惯总结

结合仓库代码和 `CursorRules`，你的习惯非常鲜明：

### 10.1 文件头信息

很多系统文件顶部都有注释头：

- Author
- Created Date
- Description
- Version
- Last Modified

这说明你重视：

- 文件身份
- 版本演进
- 历史可读性

### 10.2 明确分段

你很喜欢这种结构：

- `---- services ----`
- `---- requires ----`
- `---- common variables ----`
- `---- server variables ----`
- `---- client variables ----`
- `---- [[ server ]] ----`
- `---- [[ client ]] ----`

所以你的代码阅读方式是“按区块扫描”，不是纯粹靠函数名阅读。

### 10.3 系统字段统一

系统表经常带这些字段：

- `whiteList`
- `players`
- `tasks`
- `IsLoaded`

这已经接近你默认的“系统对象协议”。

### 10.4 双端同文件

你倾向于把同一系统的 server/client 放在一个 `init.lua` 内，通过：

- `if IsServer then ... else ... end`

来分支，而不是拆成 `server.lua / client.lua`。

### 10.5 统一系统调用，不爱散 Remote

你明显更喜欢：

- `SystemMgr.systems.XxxSystem:Method(...)`
- `self.Client:Method(...)`
- `self.Server:Method(...)`

而不是在业务里到处声明和引用零散 RemoteEvent。

### 10.6 UI 行为统一收口

按钮交互、通知、modal、timer、reward 都尽量走 `uiController`。

这说明你的设计倾向是：

- 业务逻辑系统化
- UI 交互中心化

### 10.7 风格上偏“够用、直接”

从 `CursorRules` 和代码现状一起看：

- 你不喜欢过度防御
- 更愿意相信资源路径和调用契约
- 关键路径才做保护

例如：

- 系统远程层做 `pcall`
- 玩家离场做 alive guard
- schedule 做 pcall
- 但普通业务代码里不会每行都写 nil check

### 10.8 字符串风格处于过渡态

`CursorRules` 已明确要求：

- 优先使用 Luau 反引号插值
- 少用 `string.format`

但实际仓库里仍有一些旧代码还在用 `string.format`。

所以更准确的结论是：

- 你的“目标风格”是 backtick interpolation
- “历史存量代码”还没完全统一

## 11. CursorRules 提炼结论

`.cursor/rules` 主要表达了这些原则：

### 架构层

- 新文件要放对目录
- 新系统按 `Template` 结构建立
- 配置写进 `GameConfig / Keys / Types / Presets`

### 代码风格层

- Luau 优先用反引号插值
- 不要过度防御式编程
- 不用 `do end` 代码块

### 工作流层

- 新功能先找归属系统
- 服务端逻辑优先
- UI 统一用 `uiController.SetButtonHoverAndClick`
- 周期任务用 `ScheduleModule`
- 广播用 `self.AllClients`

### MCP / UI 层

- 非运行时访问 UI 要从 `StarterGui` 查，不是 `PlayerGui`
- UI 尽量用 `Scale`
- 注意多分辨率和安全区

### 工具链层

- Excel 通过 `Excels/ExcelToLua3.py` 导出到 `src/ReplicatedStorage/ExcelConfig`

## 13. 后续开发时最值得坚持的几个关键点

- 所有跨端通信继续统一走 `SystemMgr`
- 玩家数据只让服务端权威写入
- `PlayerRemoving` 顺序不要被打乱
- 新系统优先考虑迁移到 `BaseSystem`
- UI 初始化继续维持 pendingCalls 模式或 `BaseSystem:InitUI`
- 新玩法系统只启用真正需要的系统，避免旧依赖链继续污染
